
# SOC L1 Alert Triage

## Alert Fundamentals

An alert is a notification generated by a security tool when specific events or patterns indicate potentially suspicious activity. Alerts allow analysts to focus on high-risk events instead of manually reviewing millions of raw logs.

### Event to Alert Lifecycle

1. An event occurs (user login, process execution, file download, network connection, etc.)
2. The system records the event in logs.
3. Logs are forwarded to security platforms such as SIEM, EDR, or NDR solutions.
4. Detection rules analyse incoming logs.
5. Suspicious activity generates an alert for analyst review.

Without alerting systems, SOC analysts would need to manually sift through enormous volumes of log data.

---

## SOC Roles in Alert Management

### L1 Analyst
- First line of defence.
- Reviews and triages alerts.
- Identifies false positives and true positives.
- Escalates complex incidents to L2 analysts.

### L2 Analyst
- Performs deeper investigations.
- Handles advanced threats and incident response activities.
- Provides remediation guidance.

### SOC Engineer
- Builds and maintains detection rules.
- Ensures alerts contain useful context for analysts.
- Tunes detections to reduce noise.

### SOC Manager
- Oversees alert handling processes.
- Tracks analyst performance and response times.
- Ensures incidents are managed effectively.

## Core Alert Properties

### Alert Time
- Time the alert was generated.
- May differ slightly from the actual event time.

### Alert Name
- Short description of the suspicious activity.
- Examples:
  - Unusual Login Location
  - Email Marked as Phishing
  - Potential Data Exfiltration
  - Windows Brute Force

### Alert Severity
Used to indicate urgency.

| Severity | Meaning |
|-----------|-----------|
| Low | Informational activity |
| Medium | Potentially suspicious |
| High | Likely malicious activity |
| Critical | Immediate investigation required |

### Alert Status
Tracks investigation progress.

Common statuses:
- New
- In Progress
- Closed
- Resolved

### Alert Verdict
Classification of the investigation outcome.

- True Positive (real threat)
- False Positive (benign activity)

### Alert Assignee
- Analyst responsible for the alert.
- Ownership ensures accountability throughout the investigation.

### Alert Description
Typically contains:
- Detection logic
- Why the activity is suspicious
- Investigation guidance

### Alert Fields
Additional evidence used during triage, such as:
- Username
- Hostname
- IP address
- Command line activity
- Process names

## Alert Prioritisation

Proper prioritisation ensures serious threats are investigated first.

### Step 1: Filter Alerts
Focus only on:
- New alerts
- Unassigned alerts
- Unresolved alerts

Avoid duplicating work already being performed by another analyst.

### Step 2: Prioritise by Severity

Order of investigation:

1. Critical
2. High
3. Medium
4. Low

Higher severity alerts generally represent greater business risk and higher confidence detections.

### Step 3: Prioritise by Time

Within the same severity level:

- Investigate oldest alerts first.
- Newer alerts follow afterwards.

Older unresolved incidents may already be progressing through later attack stages.

## Alert Triage Workflow

### 1. Initial Actions

Take ownership of the alert:

- Assign the alert to yourself.
- Change status to **In Progress**.
- Review alert details and supporting evidence.
- Understand what detection triggered the alert.

---

### 2. Investigation

The core analytical phase.

Key questions:

#### Who is affected?
Identify:
- User accounts
- Endpoints
- Servers
- Cloud resources
- Network assets

#### What happened?
Determine:
- Login activity
- Malware execution
- Credential misuse
- Data access
- Lateral movement

#### What happened before and after?
Review surrounding events to establish context.

Look for:
- Related alerts
- Process chains
- Network connections
- Additional suspicious behaviour

#### Validate Findings
Use:
- Threat intelligence sources
- Internal documentation
- Known attack patterns
- Previous investigations

---

### 3. Final Actions

Determine the investigation outcome.

#### True Positive
Evidence confirms malicious or suspicious activity.

Actions:
- Document findings.
- Escalate if required.
- Follow incident response procedures.

#### False Positive
Activity is legitimate and poses no threat.

Actions:
- Document reasoning.
- Close the alert.

## Investigation Documentation

Every alert should include clear analyst notes covering:

- What was investigated.
- Evidence reviewed.
- Findings discovered.
- Reasoning behind the verdict.
- Escalation actions if required.

Strong documentation ensures investigations are repeatable and understandable by other analysts.

---

## Key Takeaways

- Alerts are generated from suspicious events identified by security tools.
- L1 analysts are responsible for initial alert triage and classification.
- Alert severity and age drive investigation priority.
- Effective triage follows a structured process:
  - Assign
  - Review
  - Investigate
  - Classify
  - Document
  - Close or Escalate
- Accurate documentation and evidence-based decision making are critical skills for SOC analysts.
- The goal of alert triage is to quickly distinguish real threats from benign activity while ensuring genuine incidents receive timely attention.
