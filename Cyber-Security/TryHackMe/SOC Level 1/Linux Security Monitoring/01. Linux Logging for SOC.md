# Linux Logging for SOC

## Overview
- Linux is widely used for servers, cloud infrastructure, and containers, making Linux log analysis a critical SOC skill.
- Unlike Windows Event Viewer, Linux primarily stores logs as **plain text files** under `/var/log/`.
- Log formats are less structured than Windows Event Logs and vary between distributions.
- Runtime events (process creation, file access, network activity) are **not logged by default** and require additional auditing tools such as **auditd**.

---

# Linux Log Locations

| Log File | Purpose |
|----------|----------|
| `/var/log/syslog` | General system events (Debian/Ubuntu) |
| `/var/log/messages` | General system events (RHEL/CentOS) |
| `/var/log/auth.log` | Authentication, SSH, sudo, user management |
| `/var/log/secure` | Authentication log (RHEL/CentOS) |
| `/var/log/kern.log` | Kernel events and errors |
| `/var/log/dpkg.log` | Package installation/removal (Debian) |
| `/var/log/apt/` | APT package manager logs |
| `/var/log/dnf.log` | DNF package manager logs (RHEL) |
| `/var/log/yum.log` | Yum package manager logs |
| `/var/log/audit/audit.log` | auditd runtime monitoring events |

# Working with Logs

## View Logs

```bash
cat /var/log/syslog
less /var/log/syslog
head /var/log/syslog
tail /var/log/syslog
tail -f /var/log/syslog
```

---

## Search Logs

Filter for keywords:

```bash
grep CRON /var/log/syslog
```

Exclude matches:

```bash
grep -v CRON /var/log/syslog
```

Recursive search across all log files:

```bash
grep -R "login" /var/log
```

Multiple keywords (Extended Regex):

```bash
grep -E "auth|login|session" /var/log/auth.log
```

Case insensitive:

```bash
grep -i sshd /var/log/auth.log
```

Search for arguments beginning with "-":

```bash
grep -- "-host" /var/log/audit/audit.log
```

# Authentication Logs (`auth.log`)

Primary SOC log for:

- User logins
- SSH activity
- sudo usage
- su sessions
- Cron sessions
- User management

Location:

```text
/var/log/auth.log
```

(RHEL: `/var/log/secure`)

---

## Session Events

Successful logins/logouts:

```bash
grep -E "session opened|session closed" /var/log/auth.log
```

Can reveal:

- Local console logins
- SSH sessions
- SMB sessions
- sudo sessions
- cron jobs

---

## SSH Authentication

Find successful and failed SSH logins:

```bash
grep sshd /var/log/auth.log | grep -E "Accepted|Failed"
```

Typical format:

```
Failed password for root from 192.168.1.50
Accepted publickey for bob from 10.10.10.5
```

Useful IOC fields:

- Username
- Source IP
- Authentication method
- Port

---

## User Management Events

Search for:

```bash
grep -E "(passwd|useradd|usermod|userdel)\[" /var/log/auth.log
```

Detects:

- Password changes
- User creation
- User deletion
- Group membership changes

Common attacker persistence:

- Create hidden user
- Add to sudo group

---

## Sudo Commands

View commands executed via sudo:

```bash
grep "COMMAND=" /var/log/auth.log
```

Useful for identifying privileged activity.

---

## Bash History Limitations

Attackers can evade history by:

Leading space

```bash
 echo hidden command
```

Running scripts

```bash
./script.sh
```

Using another shell

```bash
sh
```

Also:

- Commands only saved on logout (default)
- Non-interactive processes aren't recorded

Because of these limitations, Bash history is **not considered a reliable forensic source**.

# Runtime Monitoring

Default Linux logging **does not** capture:

- Process creation
- File opens
- File modifications
- Network connections

These require runtime monitoring.

Equivalent Windows concept:

- Sysmon

Linux equivalent:

- auditd
- Sysmon for Linux
- Falco
- Osquery
- EDR

---

# System Calls

Programs cannot directly interact with hardware.

They request kernel services via **system calls**.

Common examples:

| System Call | Purpose |
|-------------|----------|
| execve | Execute program |
| open/openat | Open file |
| read | Read file |
| write | Write file |
| connect | Network connection |

Key point:

> Nearly every meaningful action on Linux eventually results in one or more system calls.

# auditd (Audit Daemon)

Purpose:

Logs selected runtime events.

Configuration:

```
/etc/audit/rules.d/
```

Logs:

```
/var/log/audit/audit.log
```

---

## Useful ausearch Commands

Readable output:

```bash
ausearch -i
```

Search by key:

```bash
ausearch -i -k proc_wget
```

Search by executable:

```bash
ausearch -x wget
```

Search by audit event ID:

```bash
ausearch -a 1527
```

---

# auditd Event Types

Common records:

| Record | Purpose |
|---------|----------|
| PROCTITLE | Full executed command |
| EXECVE | Command arguments |
| CWD | Current working directory |
| PATH | Files accessed |
| SYSCALL | Kernel system call metadata |

All records sharing the same event ID belong to one activity.

Example:

```
audit(...:1527)
```

Everything with `:1527` represents the same execution.

---

# Understanding auditd Fields

Useful fields:

| Field | Meaning |
|--------|----------|
| pid | Process ID |
| ppid | Parent Process ID |
| auid | Original logged-in user |
| uid | Effective user |
| tty | Terminal session |
| exe | Executed binary |
| key | Rule name that generated the event |

---

# Reconstructing Commands

auditd stores command arguments separately:

```
a0="./naabu"
a1="-host"
a2="192.168.50.0/24"
a3="-top-ports"
a4="100"
```

Reconstructed command:

```bash
./naabu -host 192.168.50.0/24 -top-ports 100
```

This revealed the scanned network:

```
192.168.50.0/24
```

---

# Useful Investigation Workflow

1. Find suspicious executable.

```bash
grep naabu /var/log/audit/audit.log
```

2. Locate the audit event ID.

Example:

```
audit(...:1527)
```

3. Display all records for that event.

```bash
grep "1527" /var/log/audit/audit.log
```

or

```bash
ausearch -a 1527
```

4. Read the `EXECVE` record.

5. Reconstruct the original command.

---

# Practical SOC Tips

- Start with `auth.log` for authentication investigations.
- Use `grep` to rapidly narrow large log files.
- Prefer `ausearch` over raw `grep` for auditd investigations.
- `EXECVE` usually contains the exact command-line arguments.
- `PROCTITLE` often shows the full executed command.
- Group audit records using their shared event ID.
- Bash history is useful but should never be trusted as the sole source of evidence.
- Runtime monitoring fills the visibility gaps left by standard Linux logs.
- Understanding system calls makes learning Linux EDRs and runtime monitoring tools significantly easier.
