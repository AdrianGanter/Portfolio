# Email Security and Phishing Prevention

## Overview

Email remains one of the most common attack vectors used by threat actors to gain initial access to an organization. Modern email security relies on a combination of authentication standards, technical controls, monitoring, and user awareness to reduce the risk of phishing, spoofing, malware delivery, and credential theft.

This document summarizes the core technologies and analysis techniques used to defend against phishing attacks and verify email legitimacy.

---

## Sender Policy Framework (SPF)

SPF is an email authentication mechanism used to verify whether a mail server is authorized to send email on behalf of a domain.

When an email is received, the destination mail server queries DNS for the sender domain's SPF record and checks whether the originating IP address is permitted to send mail for that domain.

### Example SPF Record

```text
v=spf1 ip4:127.0.0.1 include:_spf.google.com -all
```

### Components

* `v=spf1` – SPF version identifier
* `ip4:` – Authorized IPv4 sender
* `include:` – Trusted third-party mail services
* `-all` – Reject unauthorized senders

### Common SPF Results

| Result    | Action             |
| --------- | ------------------ |
| Pass      | Accept email       |
| Neutral   | Accept email       |
| None      | Accept email       |
| SoftFail  | Flag as suspicious |
| PermError | Flag as suspicious |
| Fail      | Reject email       |
| TempError | Reject email       |

### Security Benefit

SPF helps prevent attackers from spoofing legitimate domains by verifying sending infrastructure against DNS records.

## DomainKeys Identified Mail (DKIM)

DKIM provides email authentication through digital signatures.

When an email is sent, the mail server signs the message using a private key. The receiving server retrieves the public key from DNS and validates the signature.

### Example DKIM Record

```text
v=DKIM1; k=rsa; p=<public_key>
```

### Components

* `v=DKIM1` – DKIM version
* `k=rsa` – Cryptographic algorithm
* `p=` – Public key used for verification

### Security Benefits

* Verifies sender authenticity
* Protects message integrity
* Detects unauthorized modifications
* Survives email forwarding better than SPF

### Common Failure Indicators

* Missing public key
* Invalid signature
* DNS misconfiguration
* Email modification during transit

## S/MIME

Secure/Multipurpose Internet Mail Extensions (S/MIME) provides encryption and digital signing for email communications.

S/MIME relies on public key cryptography to ensure secure communication between parties.

### Digital Signatures

Digital signatures provide:

* Authentication
* Non-repudiation
* Data integrity

The sender signs the message using their private key and recipients verify it using the sender's public key.

### Encryption

Encryption provides:

* Confidentiality
* Secure message transmission

The sender encrypts the email using the recipient's public key, ensuring only the recipient can decrypt it using their private key.

### Security Benefits

* Protects sensitive email content
* Prevents tampering
* Verifies sender identity
* Supports secure business communications

## Email Authentication Layer

Modern email security commonly relies on three complementary technologies:

### SPF

Validates authorized sending infrastructure.

### DKIM

Validates message integrity and sender authenticity.

### DMARC

Combines SPF and DKIM results while defining organizational email handling policies.

Together these controls significantly reduce email spoofing and phishing success rates.

---

## Technical Email Security Controls

Organizations deploy multiple defensive layers to detect and block malicious email activity.

### Email Filtering

Filters messages based on:

* IP reputation
* Domain reputation
* Known malicious indicators
* Spam characteristics

### Secure Email Gateways (SEGs)

Inspect emails for:

* Impersonation attempts
* Spoofing
* Malicious attachments
* Social engineering techniques

### Link Rewriting

Suspicious URLs are rewritten and redirected through security inspection services before users access them.

Benefits include:

* Real-time URL analysis
* Delayed detonation detection
* Protection against newly weaponized links

### Sandboxing

Suspicious files and URLs are executed inside isolated virtual environments.

Benefits include:

* Malware behavior observation
* Safe execution
* IOC generation
* Payload detection

---

## User-Focused Security Controls

Technology alone cannot eliminate phishing risk. User education remains a critical defensive layer.

### Trust and Warning Indicators

Modern email platforms provide visual warnings such as:

* External Sender
* Suspicious Link
* Unverified Sender
* Newly Registered Domain

These warnings help users identify potentially dangerous emails before interacting with them.

### Phishing Reporting

Organizations should provide simple reporting mechanisms that allow users to submit suspicious emails directly to security teams.

### Security Awareness Training

Users should be trained to recognize:

* Phishing attempts
* Social engineering tactics
* Credential harvesting
* Suspicious attachments
* Malicious URLs

### Phishing Simulations

Controlled phishing campaigns help:

* Measure user susceptibility
* Reinforce training
* Identify high-risk behaviors
* Improve organizational resilience

## Email Traffic Analysis Using Wireshark

Email security investigations often involve packet capture analysis to identify suspicious SMTP communications and malicious email activity.

### SMTP Analysis

During traffic analysis I examined SMTP conversations to identify:

* Email transmission attempts
* Mail server responses
* Delivery failures
* Rejected messages
* Spam filtering activity

Useful filters included:

```text
smtp
```

and

```text
smtp.response.code
```

These filters helped isolate SMTP server responses and identify suspicious activity.

### Reviewing SMTP Responses

SMTP response codes provide insight into how mail servers process incoming messages.

Examples include:

* 220 Service Ready
* 250 Requested Action Completed
* 550 Mailbox Unavailable
* 552 Message Rejected

Reviewing response codes can reveal:

* Reputation-based blocking
* Spam filtering decisions
* Mail delivery failures
* Security policy enforcement

---

## Internet Message Format (IMF) Analysis

Using Wireshark IMF filtering allowed examination of individual email contents within SMTP traffic.

Useful filter:

```text
imf
```

This enabled inspection of:

* Sender addresses
* Recipient addresses
* Subject lines
* Content types
* Attachments
* Encoding methods
* Email client metadata

### Attachment Analysis

During packet analysis I reviewed attachment metadata including:

* Attachment filenames
* MIME content types
* Transfer encoding methods
* Potentially suspicious file extensions

Examples of indicators that may warrant further investigation:

* Executable attachments
* Screensavers (.scr)
* Macro-enabled documents
* Encoded payloads
* Double file extensions

## Key Takeaways

* SPF validates authorized mail servers.
* DKIM verifies email authenticity and integrity.
* S/MIME provides encryption and digital signatures.
* DMARC strengthens email authentication policies.
* Sandboxing enables safe malware analysis.
* Secure Email Gateways provide additional filtering and inspection.
* User awareness remains a critical defense against phishing.
* Wireshark can be used to analyze SMTP traffic, delivery failures, attachments, and suspicious email behavior.
* Multiple security controls working together provide the strongest protection against phishing attacks.
