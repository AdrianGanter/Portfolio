# Snort

## Overview

**Snort** is an open-source, rule-based **Network Intrusion Detection System (IDS)** and **Intrusion Prevention System (IPS)** developed by Martin Roesch and maintained by Cisco Talos.

It inspects network traffic against a set of rules and generates alerts (or blocks traffic in IPS mode) when matches occur. :contentReference[oaicite:0]{index=0}

---

# How Snort Works

```
Packets
   │
Packet Decoder
   │
Preprocessors
   │
Detection Engine (Rules)
   │
Logging / Alerting
   │
Output Plugins
```

### Components

- **Packet Decoder** – Captures packets.
- **Preprocessors** – Normalises and prepares packets.
- **Detection Engine** – Compares traffic against Snort rules.
- **Logging & Alerting** – Generates alerts and logs.
- **Output Plugins** – Sends output to console, files, syslog, databases, etc. :contentReference[oaicite:1]{index=1}

---

# IDS vs IPS

## IDS (Intrusion Detection System)

- Passive
- Detects suspicious traffic
- Generates alerts only

## IPS (Intrusion Prevention System)

- Inline with network traffic
- Detects **and blocks** malicious traffic
- Requires inline mode (`-Q`, DAQ)

# Basic Rule Structure

<img width="1140" height="784" alt="image" src="https://github.com/user-attachments/assets/50b9f365-94fa-476a-98d2-71c9d9b0de07" />

```
ACTION PROTOCOL SRC_IP SRC_PORT DIRECTION DST_IP DST_PORT (OPTIONS)
```

Example:

```snort
alert icmp any any <> any any (msg:"ICMP Packet Found"; sid:100001; rev:1;)
```

Meaning:

- alert → generate alert
- icmp → protocol
- any any → any source
- <> → both directions
- msg → alert text
- sid → unique rule ID
- rev → rule revision

# Snort Operating Modes

## Sniffer Mode

Simply displays packets.

Useful for quick packet inspection.

---

## Packet Logger Mode

Logs packets to disk.

Useful for recording traffic for later analysis.

---

## IDS / Detection Mode

Uses rules to inspect packets.

Produces alerts when rules match.

Most common operating mode.

---

## IPS / Inline Mode

Blocks traffic matching rules.

Requires:

```bash
-Q
--daq afpacket
```

or appropriate DAQ configuration.

# Essential Linux Commands

## Test Configuration

Always do this before running Snort.

```bash
sudo snort -c /etc/snort/snort.conf -T
```

Checks:

- syntax
- configuration
- rules

---

## Quiet Mode

Hide startup banner.

```bash
-q
```

---

## Read a PCAP

```bash
sudo snort -c /etc/snort/snort.conf -r capture.pcap
```

---

## Read Multiple PCAPs

```bash
sudo snort -c /etc/snort/snort.conf --pcap-list="one.pcap two.pcap"
```

---

## Specify Config File

```bash
-c /etc/snort/snort.conf
```

---

## Log Output Directory

```bash
-l .
```

Logs to current directory.

---

## Disable Logging

```bash
-N
```

Useful when only console output is required.

---

## Background (Daemon) Mode

```bash
sudo snort -c /etc/snort/snort.conf -D
```

Runs Snort in the background.

Check process:

```bash
ps -ef | grep snort
```

Stop process:

```bash
sudo kill -9 PID
```

# Alert Modes

## Console

```bash
-A console
```

Shows alerts directly on screen.

Fast summary format.

---

## Fast

```bash
-A fast
```

Creates alert file containing:

- timestamp
- source IP
- destination IP
- ports
- alert message

Quick overview.

---

## Full

```bash
-A full
```

Provides complete alert information.

Best for investigations.

---

## CMG

```bash
-A cmg
```

Displays:

- packet headers
- payload
- hex dump
- ASCII

Useful when analysing packet contents.

---

## None

```bash
-A none
```

No alerts generated.

Only packet logs are created.

Useful when collecting traffic without alert noise.

# Useful Snort Commands

Run on live traffic:

```bash
sudo snort -c /etc/snort/snort.conf
```

Test config:

```bash
sudo snort -c /etc/snort/snort.conf -T
```

Read PCAP:

```bash
sudo snort -c /etc/snort/snort.conf -r file.pcap
```

Read PCAP and log locally:

```bash
sudo snort -c /etc/snort/snort.conf -A full -l . -r file.pcap
```

Run with only local rules:

```bash
sudo snort -c /etc/snort/rules/local.rules -A console
```

Background mode:

```bash
sudo snort -c /etc/snort/snort.conf -D
```

IPS mode:

```bash
sudo snort -Q --daq afpacket
```

# Editing Rules

Edit custom rules:

```bash
sudo gedit /etc/snort/rules/local.rules
```

Good practice:

- Comment out old rules instead of deleting them.
- Test rules before production.
- Increase `rev` when changing a rule.
- Keep backups of configuration files.

---

# Rule Writing Tips

Start simple.

Instead of creating one huge rule:

1. Create basic rule.
2. Test.
3. Add one option.
4. Test again.
5. Repeat.

Much easier to troubleshoot syntax errors.

# Practical Rule Examples

Alert on all ICMP:

```snort
alert icmp any any <> any any (msg:"ICMP"; sid:100001; rev:1;)
```

Match SYN packets:

```snort
flags:S;
```

Match PSH+ACK:

```snort
flags:PA;
```

Match UDP packets with same source/destination IP:

```snort
ip_proto:17;
```

Match specific IP ID:

```snort
id:35369;
```

These examples were used throughout the exercises to identify specific traffic patterns.

# Best Practices

- Validate configuration before every run (`-T`).
- Use `local.rules` for custom signatures.
- Never overwrite working rules—comment them out instead.
- Increment `rev` after modifying rules.
- Test rules on PCAPs before deploying.
- Build rules incrementally.
- Keep backups of `snort.conf`.
- Prefer modifying existing rules over reinventing them.

# Key Takeaways

- Snort is fundamentally **a rule engine**.
- The **quality of your rules determines detection quality**.
- **Always test** (`-T`) before running.
- Learn to recognise when to use **Console**, **Fast**, **Full**, **CMG**, and **None** alert modes.
- **`snort.conf` controls the engine; `local.rules` controls your custom detections.**
- Becoming proficient with Snort is less about memorising commands and more about understanding how traffic flows through **Decoder → Preprocessors → Detection Engine → Alerts**.
