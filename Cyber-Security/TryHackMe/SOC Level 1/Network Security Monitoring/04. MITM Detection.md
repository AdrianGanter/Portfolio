# Man-in-the-Middle (MITM) Detection

## Overview
A Man-in-the-Middle (MITM) attack occurs when an attacker secretly positions themselves between two communicating parties to intercept, monitor, modify, or redirect traffic. Successful MITM attacks often lead to credential theft, session hijacking, malware delivery, or data exfiltration.

---

# MITM Attack Flow

1. Intercept network traffic.
2. Trick victims into sending traffic through the attacker.
3. Read or modify communications.
4. Forward traffic to the legitimate destination.
5. Victim remains unaware.

---

# Common MITM Techniques

## ARP Spoofing
Purpose:
- Become the gateway for local network traffic.

How it works:
- Attacker sends forged ARP replies.
- Victim associates the gateway IP with the attacker's MAC.
- All traffic flows through the attacker.

Indicators:
- Duplicate IP → MAC mappings.
- Multiple MACs claiming the same IP.
- Large number of unsolicited ("gratuitous") ARP replies.
- Sudden increase in ARP traffic.
- Gateway IP advertised by an unknown MAC.

---

## DNS Spoofing (DNS Cache Poisoning)

Purpose:
- Redirect victims to attacker-controlled systems.

How it works:
- Victim requests a domain.
- Attacker replies before the legitimate DNS server.
- Victim caches the fake IP.
- User connects directly to attacker.

Indicators:
- Multiple DNS responses for one query.
- DNS replies from unexpected IP addresses.
- Very low TTL values.
- DNS responses without matching requests.
- Internal hosts acting as DNS servers.

---

## SSL Stripping

Purpose:
- Downgrade HTTPS to HTTP so credentials are sent in plaintext.

How it works:
1. Victim requests HTTPS.
2. Attacker intercepts traffic.
3. Attacker maintains HTTPS with the real server.
4. Victim communicates with attacker over HTTP.
5. Credentials become visible.

Indicators:
- HTTPS suddenly changes to HTTP.
- TLS handshakes disappear.
- Login credentials visible in HTTP POST requests.
- Persistent redirects to HTTP.

---

# MITRE ATT&CK / Kill Chain

MITM commonly appears during:

- **Exploitation**
  - Attacker gains interception capability.

- **Installation**
  - Injects malware or establishes persistence.

Often enables:
- Credential theft
- Session hijacking
- Malware delivery
- Data exfiltration

# ARP Detection in Wireshark

## Show all ARP traffic

```wireshark
arp
```

---

## ARP Requests

```wireshark
arp.opcode == 1
```

Shows:
- "Who has..." requests.

---

## ARP Replies

```wireshark
arp.opcode == 2
```

Look for:
- Numerous unsolicited "is-at" replies.

---

## Gratuitous ARP

```wireshark
arp.isgratuitous
```

Strong indicator of ARP poisoning.

---

## Legitimate Gateway

```wireshark
arp &&
arp.src.proto_ipv4 == 192.168.10.1 &&
eth.src == 02:aa:bb:cc:00:01
```

Shows legitimate gateway advertisements.

---

## Gateway IP Claimed by Any MAC

```wireshark
arp.opcode == 2 &&
arp.src.proto_ipv4 == 192.168.10.1
```

Used to identify fake MAC addresses.

---

## ARP Responses Advertising Gateway

```wireshark
arp.opcode == 2 &&
_ws.col.info contains "192.168.10.1 is at"
```

Quickly reveals spoofed gateway advertisements.

---

## Duplicate Address Detection

```wireshark
arp.duplicate-address-detected ||
arp.duplicate-address-frame
```

Shows conflicting IP-to-MAC mappings.

---

# DNS Spoofing Detection

## Show DNS Traffic

```wireshark
dns
```

---

## DNS Responses

```wireshark
dns.flags.response == 1
```

---

## Legitimate DNS Server

```wireshark
dns.flags.response == 1 &&
ip.src == 8.8.8.8
```

Baseline legitimate responses.

---

## Specific Domain

```wireshark
dns &&
dns.qry.name == "corp-login.acme-corp.local"
```

Shows all requests/responses for the target domain.

---

## Legitimate DNS Replies

```wireshark
dns.flags.response == 1 &&
ip.src == 8.8.8.8 &&
dns.qry.name == "corp-login.acme-corp.local"
```

---

## Rogue DNS Replies

```wireshark
dns.flags.response == 1 &&
ip.src != 8.8.8.8 &&
dns.qry.name == "corp-login.acme-corp.local"
```

Excellent filter for identifying DNS spoofing.

---

# SSL Stripping Detection

## Show TLS Traffic

```wireshark
tls || ssl
```

---

## TLS Handshakes

```wireshark
tls.handshake.type == 1 &&
tls.handshake.extensions_server_name == "corp-login.acme-corp.local"
```

Confirms the website normally uses HTTPS.

---

## DNS Spoofing by Attacker

```wireshark
dns.flags.response == 1 &&
ip.src == 192.168.10.55 &&
dns.qry.name == "corp-login.acme-corp.local"
```

Shows attacker redirecting victims.

---

## HTTP Traffic After Stripping

```wireshark
http &&
ip.src == 192.168.10.10 &&
ip.dst == 192.168.10.55
```

Shows victim communicating over HTTP instead of HTTPS.

Useful for identifying:
- Login requests
- Credentials
- HTTP POST bodies

---

# Investigation Workflow

1. Filter ARP traffic.
2. Identify unsolicited ARP replies.
3. Confirm gateway IP is mapped to multiple MAC addresses.
4. Inspect DNS traffic.
5. Identify rogue DNS responses.
6. Confirm victim redirected to attacker IP.
7. Verify TLS disappears.
8. Inspect HTTP traffic.
9. Recover credentials from plaintext POST requests.

---

# Key Indicators of MITM

### ARP Spoofing
- Duplicate IP/MAC mappings.
- Gratuitous ARP replies.
- Unknown MAC claiming gateway IP.

### DNS Spoofing
- Multiple replies for one query.
- Responses from unexpected hosts.
- Very low TTL.
- Internal rogue DNS responder.

### SSL Stripping
- HTTPS downgraded to HTTP.
- Missing TLS handshakes.
- Credentials visible in plaintext.
- HTTP POST requests containing usernames/passwords.

---

# Practical Summary

Performed MITM investigation by:

- Detecting ARP cache poisoning through duplicate IP-to-MAC mappings.
- Identifying gratuitous ARP replies from the attacker.
- Confirming gateway impersonation.
- Detecting forged DNS responses redirecting victims.
- Identifying rogue DNS servers.
- Confirming SSL stripping by observing HTTPS downgraded to HTTP.
- Recovering plaintext credentials from HTTP POST requests.
- Correlating ARP Spoofing → DNS Spoofing → SSL Stripping into a complete multi-stage MITM attack.
