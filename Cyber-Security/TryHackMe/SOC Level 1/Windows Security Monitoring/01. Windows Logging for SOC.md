# Windows Logging for Security Monitoring

## Why Windows Logging Matters

Windows logs provide a record of user and system activity, making them essential for:

- Incident response
- Threat hunting
- Alert triage
- Attack timeline reconstruction

Every logged action is stored as an **event**.

---

# Windows Event Logs

## Log Storage

- Stored as binary **EVTX** files
- Location:

```text
C:\Windows\System32\winevt\Logs
```

Each EVTX file represents a different log source.

Examples:

- Application
- Security
- System
- Sysmon (if installed)

---

# Windows Security Log

The **Security log** is the most valuable default Windows log for defenders.

---

# Authentication Events

## Event ID 4624 — Successful Logon

Purpose:

- Detect successful authentication
- Trace attacker entry point
- Correlate later activity

### Important Fields

- Username
- Source IP
- Workstation Name
- Logon Type
- Logon ID

---

## Event ID 4625 — Failed Logon

Purpose:

- Detect brute force
- Detect password spraying
- Detect login scanning

---

# Logon Types

| Type | Meaning |
|------|---------|
| 2 | Interactive (local keyboard) |
| 3 | Network login |
| 10 | Remote Desktop (RDP) |

These are the most commonly encountered during investigations.

---

# Brute Force Detection

Filter:

```text
Event ID = 4625
```

Watch for:

- Many failed logins
- Same account repeatedly targeted
- Multiple usernames attempted
- Unknown workstation names
- Suspicious source IPs

Example:

```
Administrator
admin
helpdesk
backup
```

Large numbers of different usernames often indicate **password spraying**.

Repeated failures against one account usually indicate **brute force**.

---

# Successful Login Investigation

Filter:

```text
Event ID = 4624
```

Focus on:

- Logon Type 10 (RDP)
- Source IP
- Workstation
- Username

Always record the:

> **Logon ID**

This uniquely identifies the login session and allows correlation with future events.

---

# Logon ID

One of the most important fields.

Used to correlate:

- Login
- Process creation
- User creation
- Privilege changes
- Later attacker activity

Think of it as the attacker's session identifier.

# Investigating User Management

Every event contains three parts:

## Subject

Who performed the action.

Contains:

- Username
- Logon ID

---

## Object

Who was affected.

Examples:

- New user
- Existing user
- Group member

---

## Details

Exactly what changed.

Examples:

- Group added
- Password reset
- Account enabled

---

# Red Flags

Unexpected:

- New local users
- Weekend changes
- Unknown admin accounts
- Accounts added to Administrators
- Password resets without approval

Common attacker behaviour:

- Create backdoor account
- Add account to Administrators
- Reset privileged passwords
- Enable disabled accounts

---

# Process Creation Logging

## Security Event 4688

Purpose:

Logs every new process.

Limitations:

- Disabled by default
- Basic information only

---

# Sysmon

Recommended over native process logging.

Benefits:

- Better visibility
- More fields
- Configurable
- Industry standard

Logs found at:

```text
Applications and Services Logs
→ Microsoft
→ Windows
→ Sysmon
→ Operational
```

---

# Sysmon Event ID 1

Process Creation

Contains:

## Process Information

- Process name
- Executable path
- Command line

---

## Parent Process

Shows what launched the process.

Useful for building process trees.

Example:

```
explorer.exe
   └── powershell.exe
        └── cmd.exe
```

---

## Binary Information

Includes:

- SHA256 hash
- MD5 hash
- Digital signature
- Company metadata

Useful for:

- VirusTotal lookups
- Malware validation

---

## User Context

Includes:

- Username
- Logon ID

Allows correlation back to login events.

---

# Process Investigation

Red flags:

- Executables in:

```
C:\Temp
C:\Users\Public
```

- Random filenames

Examples:

```
aa.exe
jqyvpqldou.exe
```

- Suspicious parent processes

Example:

```
notepad.exe
    ↓
cmd.exe
```

Notepad should not launch command shells.

---

# Process Tree Analysis

Follow parent-child relationships:

```
ParentProcessID
↓

ProcessID
↓

Child Process
```

Walk backwards until the original process is identified.

# Network Investigation

Red flags:

- External IPs
- Port 80 to unusual destinations
- Port 4444
- Known malicious domains
- Suspicious DNS lookups

---

# File Investigation

Look for files created in:

```
C:\Temp
C:\Users\Public
```

Especially:

- .exe
- .bat
- .ps1
- .com

# Registry Investigation

Watch for persistence mechanisms such as:

- Run keys
- Startup registry modifications

---

# Correlation Workflow

A common investigation flow:

```
4624 Login
      ↓
Logon ID
      ↓
Sysmon Event 1
      ↓
Process ID
      ↓
Sysmon Events
      ↓
Network
      ↓
Files
      ↓
Registry
```

This reconstructs the complete attack chain.

---

# PowerShell Monitoring

Sysmon only records:

```
powershell.exe
```

It does **not** capture every command executed inside the PowerShell session.

---

# PowerShell History

Location:

```text
C:\Users\<USER>\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

Contains every command entered interactively.

---

# PowerShell History Characteristics

Records:

- Commands entered
- Commands persist after reboot
- Separate history for each user

Does **not** record:

- Command output
- Script contents
- Non-interactive execution

---

# Common Malicious PowerShell Commands

Examples:

```powershell
Get-ChildItem
```

Directory listing.

```powershell
Get-Content secrets.txt
```

Reads sensitive files.

```powershell
Get-LocalUser
```

Enumerates users.

```powershell
Get-LocalGroup
```

Enumerates groups.

```powershell
Invoke-WebRequest http://...
```

Downloads malware.

---

# Core Investigation Workflow

1. Review Security logs.
2. Identify suspicious login.
3. Save the Logon ID.
4. Find related process creation events.
5. Follow Process IDs through Sysmon.
6. Review:
   - Network connections
   - File creation
   - Registry modifications
7. Check PowerShell history if PowerShell was used.
8. Reconstruct the full attack timeline.

# Key Takeaways

- Security logs are the primary Windows log source for authentication monitoring.
- **4624** and **4625** are the foundation of login investigations.
- Always save the **Logon ID** to correlate attacker activity.
- Sysmon provides far richer visibility than native Windows logging.
- Build attack timelines by correlating **Logon ID**, **Process ID**, and Sysmon events.
- Monitor PowerShell history to uncover attacker commands that process logs alone cannot reveal.
